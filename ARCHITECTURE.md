# معمارية مشروع «الباقيات الصالحات»

بنية معيارية (Modular) بلا أدوات بناء — تعمل مباشرة على أي استضافة ثابتة. الهدف: **عزل كل مسؤولية في مكان واحد**، فأي إضافة أو تعديل مستقبلي لا يؤثّر على بقية الأجزاء.

## شجرة الملفات

```
├── index.html            # الهيكل فقط (بلا أنماط أو منطق مضمّن)
├── manifest.json         # بيانات التطبيق للتثبيت (PWA)
├── sw.js                 # عامل الخدمة: تثبيت + تخزين مؤقت + تحديث سلس
├── netlify.toml / _headers# ترويسات الأمان عند النشر
├── styles/
│   ├── tokens.css        # الرموز التصميمية (ألوان، الوضع الليلي) — هوية التطبيق
│   └── main.css          # الأنماط الأساسية والمكوّنات والأقسام
├── assets/               # اللوغو والأيقونات
└── js/
    ├── main.js           # نقطة الإقلاع: يجمع الوحدات ويربطها ويشغّلها
    ├── config.js         # الإعدادات المركزية: عناوين الخدمات، الأسماء، المفاتيح
    ├── core.js           # أدوات مشتركة: DOM، تخزين، تنبيه، أرقام، شبكة
    ├── state.js          # الحالة المشتركة للمصحف
    ├── data/             # البيانات الثابتة (معزولة عن المنطق)
    │   ├── surahs.js     # قائمة السور + حساب رقم الآية العام
    │   ├── reciters.js   # القرّاء
    │   └── content.js    # الأذكار وأدعية الميت وعبارات السبحة
    ├── ui/               # عناصر الواجهة العامة
    │   ├── theme.js      # الوضع الليلي/النهاري
    │   ├── nav.js        # التنقّل بين الأقسام (عبر خطّافات hooks)
    │   └── dedication.js # مستطيل الإهداء (مرة واحدة)
    └── features/         # الأقسام — كل ملف مستقل بذاته
        ├── player.js     # محرّك التلاوة الصوتية
        ├── quran.js      # المصحف: تحميل، عرض، بحث، حفظ
        ├── tafsir.js     # التفسير
        ├── prayer.js     # أوقات الصلاة
        ├── adhkar.js     # الأذكار وأدعية الميت
        ├── tasbih.js     # السبحة
        ├── share.js      # المشاركة
        └── install.js    # التثبيت + الإحصاءات
```

## المبادئ الهندسية

- **مسؤولية واحدة لكل وحدة (Single Responsibility):** كل ميزة في ملف مستقل داخل `features/`.
- **اعتماديات صريحة (Explicit Imports):** كل وحدة تستورد ما تحتاجه فقط عبر `import`، فلا متغيّرات عامة مخفية.
- **فصل الطبقات:** بيانات (`data/`) ← منطق (`features/`) ← واجهة (`ui/`) ← إعدادات (`config.js`).
- **تنقّل غير مقترن (Decoupled Nav):** `nav.js` لا يعرف تفاصيل الأقسام؛ تُسجّل `main.js` خطّافات التهيئة، فإضافة قسم جديد لا تمسّ التنقّل.
- **مصدر واحد للإعدادات:** تغيير عنوان خدمة أو اسم أو مفتاح إحصاء يتم من `config.js` فقط.
- **بلا دورات اعتماد (No Cycles):** `quran → player/tafsir`، و`player`/`tafsir` لا يستوردان `quran`.

## كيف تُضيف قسماً جديداً (مثال)

1. أنشئ `js/features/newfeature.js` يصدّر `initNewFeature()`.
2. أضف قسمه في `index.html`: `<section class="view" id="view-newfeature">…</section>` وزرّاً في شريط التبويبات `data-view="newfeature"`.
3. في `main.js`: استورد `initNewFeature` واستدعِه، وإن كان يحتاج تهيئة عند فتحه: `registerViewHook('newfeature', initNewFeature)`، وأضف اسمه إلى `VALID_VIEWS`.
4. أضف ملفّه إلى قائمة `SHELL` في `sw.js` وارفع رقم النسخة (`CACHE`).

لن يتأثّر أي قسم آخر بهذه الإضافة.

## تعديل المظهر
- الألوان والوضع الليلي: `styles/tokens.css`.
- المكوّنات والأقسام: `styles/main.css`.

## النشر
ارفع كامل المجلد كما هو إلى Netlify أو GitHub Pages. لا حاجة لأي خطوة بناء — المتصفح يحمّل الوحدات مباشرة عبر `type="module"`.

> عند كل تحديث: ارفع رقم النسخة في `sw.js` (مثلاً `sadaqah-v8` ← `v9`) ليصل التحديث للمستخدمين تلقائياً.

## خطوة البناء (للتوافق الشامل)
لأن بعض المتصفحات المصغّرة (داخل واتساب/تيليجرام) لا تشغّل وحدات ES، يولّد `src/build.py`
من هذه الوحدات ملفَّ `index.html` واحداً مكتفياً بذاته (أنماط + كود كلاسيكي مدمج) يعمل في كل مكان.
- طوّر دائماً في `src/` (وحدات منظّمة).
- بعد أي تعديل: `python3 src/build.py` لإعادة توليد ملف النشر.
- ملف `src/index.html` هو نسخة التطوير (تستخدم الوحدات مباشرة عبر type="module").
